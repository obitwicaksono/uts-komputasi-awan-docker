BISA MEMILIH API SECARA OTOMATIS?
================================================================================

JAWAB: HANYA SEBAGIAN

Terjadi:
  1. Docker DNS return kedua API IPs → Pilihan tersedia
  2. Client connect ke SATU IP (random selection)
  3. Koneksi di-CACHE dan REUSE untuk semua requests
  4. Hasilnya: STICKY ROUTING - requests ke instance yang sama

Dibutuhkan:
  • Load Balancer untuk intelligent routing
  • Nginx atau Traefik untuk distribute traffic evenly

================================================================================
LOAD BALANCER KEBUTUHAN
================================================================================

MENGAPA DIPERLUKAN:

1. Traffic Distribution
   └─ Nginx: Round-robin untuk merata-kan load
   
2. Automatic Failover
   └─ Jika API-1 down → traffic otomatis ke API-2
   
3. Connection Management
   └─ Multiple connections managed by LB to all instances

SOLUSI: NGINX (Simple & Production-Ready)

Config:
  upstream api_backend {
      server api:8080;  # Docker resolves to all instances
  }
  
  location / {
      proxy_pass http://api_backend;
  }

Update docker-compose.yml:
  web:
    environment:
      API_URL: http://nginx:80  # Changed from http://api:8080

Keuntungan:
  ✓ Lightweight (alpine image)
  ✓ Simple configuration
  ✓ Production-ready
  ✓ Works with any scaling level

================================================================================
LEARNING POINTS
================================================================================

1. --scale ≠ Auto Load Balancing
   └─ Creates containers but doesn't distribute traffic

2. DNS Resolution Insufficient
   └─ Need intelligent routing layer

3. Connection Pooling is Default
   └─ HTTP clients cache connections naturally

4. Stateless Design is KEY
   └─ For horizontal scaling

5. Load Balancer is Essential
   └─ For production multi-instance deployments
